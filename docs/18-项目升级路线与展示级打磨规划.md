## 3. 升级项详细清单（5 项 P0 + 3 项 P1）

### 3.1 P0-1：管理员密码认证

**当前问题表现**：`Admin/Login` 只接受 username，Service.ValidateAdminAsync 只查 `Username == username && IsAdmin == true`，无任何密码校验。

**为什么拉低成熟度**：最直接的课堂项目标志。真实系统不可能无密码登录管理端。

**面试展示价值**：高。可以在简历中写"基于 Session + 密码哈希的管理员认证体系"，演示时可展示：输入错误密码 → 页面提示 → 登录失败 → 正确密码 → 跳转管理端。

**商业化价值**：管理端安全是基本底线。无密码意味着任何人都可以管理座位/预约数据。

**AI 可参与的工作点**：
- 生成 BCrypt/ASP.NET Core Identity PasswordHasher 哈希存储代码
- 修改 Admin Login POST 添加密码验证
- 修改种子数据添加密码哈希列
- 生成密码修改功能

**建议开发步骤**：
1. User 模型添加 `PasswordHash` 列（nullable，不影响学生账号）
2. Admin Login 表单增加密码输入框
3. Service.ValidateAdminAsync 添加 `VerifyPassword` 逻辑
4. 种子数据为 admin 生成固定密码哈希（如 `admin123`）
5. 修改 seed 数据为 admin 设置密码
6. 前端展示"密码错误"提示，区分"用户不存在"和"密码错误"

**关键产物**：
- User.cs 添加 PasswordHash 属性
- AdminController.Login POST 密码验证
- 种子数据预置密码
- 测试增加"错误密码登录失败"用例

**最低验收线**：
- 输入空密码或错误密码 → 页面提示"密码错误"（不暴露用户是否存在）
- 输入正确密码 → 登录成功跳转管理端
- 管理员退出后 Session 清理，重新访问需再次输入密码
- 集成测试覆盖：密码正确 / 密码错误 / 空密码

### 3.2 P0-2：JavaScript 统一提取到 site.js

**当前问题表现**：
- `wwwroot/js/site.js` 只有 ASP.NET 默认注释，4 行空文件
- Bootstrap `needs-validation` 初始化在 Admin/Login、Admin/SeatCreate、Admin/SeatEdit 中重复了 3 次
- Reserve.cshtml 有 30+ 行 AJAX 提交逻辑
- MyReservations.cshtml 有独立的 AJAX 取消逻辑
- Admin/Seats.cshtml 有行内切换 AJAX
- 无全局 AJAX 错误处理（MyReservations 取消的 `.fail()` 完全缺失）

**为什么拉低成熟度**：面试官看到 `site.js` 是空的、每个 View 开头是 `<script>` 大段 jQuery 代码，代码组织意识直接暴露。

**面试展示价值**：高。"你把 AJAX 错误处理、401 重定向、toast 都抽象到 site.js 了"是很好的工程话术。

**商业化价值**：全局错误处理直接决定了用户能否感知到系统异常，而不是静默失败。

**AI 可参与的工作点**：
- 生成 site.js 框架（toast 工具函数、AJAX 全局错误处理、Bootstrap validation 初始化器）
- 分析每个 View 的 `<script>` 块，提取可复用部分
- 将 URL 替换为 `data-*` 属性 + JS 读取

**建议开发步骤**：
1. 创建 `site.js` 函数：`showToast(type, message)`、`initBootstrapValidation()`、`setupAjaxGlobal()` 
2. `setupAjaxGlobal` 统一处理 401（重定向到登录页）和 500（显示通用错误）
3. 将 Reserve.cshtml 的 AJAX 逻辑改为调用 site.js 中的函数
4. 将 MyReservations.cshtml 的 AJAX 取消逻辑同样抽取
5. 将重复的 `needs-validation` 初始化替换为 `initBootstrapValidation()`
6. 删除 View 中的内联 `<script>` 块

**关键产物**：
- `site.js` 完整实现（~80-120 行）
- 5 个 View 中的内联 JS 全部清理
- 每个 POST 按钮有 `data-*` 属性驱动

**最低验收线**：
- `site.js` 不为空，包含至少 showToast、initBootstrapValidation、全局 AJAX error handler
- 所有 View 不再有重复的 validation 初始化代码
- 所有 AJAX 调用在失败时显示用户可见的 toast 提示
- 401 场景（Session 过期）：全局自动跳转登录页

### 3.3 P0-3：全局异常处理 + ILogger 注入

**当前问题表现**：
- Controllers 和 Services 中 **0 个 try-catch** 块
- **0 处 ILogger 注入**（整个代码库没有一行 `ILogger<T>`）
- AJAX POST（Reserve、Cancel、SeatToggle）如果 Service 层抛异常，用户端看到的是浏览器默认错误，或者返回 500 但不显示任何提示
- 非开发环境用 `UseExceptionHandler("/Home/Error")`，但 Error.cshtml 只显示 RequestId，没有中文提示

**为什么拉低成熟度**：面试问"系统上线后用户说操作失败了，你怎么排查？"——没有日志就无法回答。这是基础设施级别的缺失。

**面试展示价值**：中高。可以说"全局异常处理中间件记录结构化日志，AJAX 返回 JSON 错误，页面请求展示友好错误页"。

**商业化价值**：高。没有日志=没有生产运维能力。

**AI 可参与的工作点**：
- 生成全局异常处理中间件代码
- 为 4 个 Service 和 2 个 Controller 添加 ILogger 注入模板
- 生成 AJAX 友好的 JSON 错误响应
- 设计 Error.cshtml 的中文友好版本

**建议开发步骤**：
1. 创建 `Middleware/ExceptionHandlingMiddleware.cs`：捕获异常 → 写日志 → 判断 AJAX 请求返 JSON / 非 AJAX 跳转错误页
2. 在 Program.cs 注册中间件，替换原有的 `UseExceptionHandler`
3. 为 HomeController、AdminController、4 个 Service、AdminAuthFilter 注入 `ILogger<T>`
4. 在关键路径添加日志：DB 操作前后、冲突检测失败、认证拦截
5. 修改 Error.cshtml：中文标题、友好描述、区分"系统错误"和"页面未找到"
6. 测试：模拟 Service 层异常验证日志写入

**关键产物**：
- `Middleware/ExceptionHandlingMiddleware.cs`
- Program.cs 中注册
- 6+ 处 ILogger 注入（2 Controller + 4 Service + AdminAuthFilter）
- 更新后的 Error.cshtml（中文）

**最低验收线**：
- 发起一个导致异常的请求（如删除运行中的 DB），非 AJAX → 显示中文错误页，AJAX → 收到 `{ success: false, msg: "服务器内部错误" }`
- 日志文件/控制台能看到异常堆栈和请求路径
- 集成测试覆盖 AJAX 和非 AJAX 两种异常场景

### 3.4 P0-4：全局 `[ValidateAntiForgeryToken]` + CSRF 防护

**当前问题表现**：
- Reserve POST、Cancel POST、SeatCreate POST、SeatEdit POST、SeatToggle POST 均无 `[ValidateAntiForgeryToken]`
- 表单中也没有 `@Html.AntiForgeryToken()` 或 `@inject Microsoft.AspNetCore.Antiforgery.IAntiforgery`
- AJAX POST 没有在请求头携带 `RequestVerificationToken`

**为什么拉低成熟度**：这是 ASP.NET Core 内置的安全机制，不启用会被视为缺乏基本安全意识。

**面试展示价值**：中。属于"安全基础"但不是最亮眼的部分。然而不提这个会有减分。

**商业化价值**：高。CSRF 是 OWASP Top 10，生产系统必须防护。

**AI 可参与的工作点**：
- 识别所有需要添加 token 的 Form 和 AJAX 调用
- 为 AJAX 生成全局 antiforgery token 设置代码
- 生成集成测试验证 CSRF 防护生效

**建议开发步骤**：
1. 在所有 `<form>` 中添加 `@Html.AntiForgeryToken()`（Admin/Login、Admin/SeatCreate、Admin/SeatEdit、Home/Reserve）
2. 在 site.js 的 `setupAjaxGlobal()` 中添加：从 `meta` 标签或 cookie 读取 antiforgery token，附加到所有 POST 请求头
3. 在 _Layout.cshtml 和 _AdminLayout.cshtml 中添加 `@inject Microsoft.AspNetCore.Antiforgery.IAntiforgery _antiforgery` + `<meta name="csrf-token" content="@_antiforgery.GetAndStoreTokens(Context).RequestToken" />`
4. 为所有 POST Action 添加 `[ValidateAntiForgeryToken]`
5. 测试：提交无 token 的 POST → 400 Bad Request

**关键产物**：
- 5 个 Form 添加 `@Html.AntiForgeryToken()`
- 2 个 Layout 添加 CSRF meta 标签
- site.js 添加全局 token 附加逻辑
- 5 个 Action 添加 `[ValidateAntiForgeryToken]`

**最低验收线**：
- 所有 POST 表单正常提交（有 token）
- 手动移除 token 后提交 → 返回 400
- AJAX POST 自动携带 token

**面试展示价值**：高。能说出"我重构了测试基础设施，消除了重复，用数据查找代替硬编码 ID，添加了关键路径的正向测试" —— 这是工程能力的体现。

**建议开发步骤**：

**关键产物**：
- 重构后的测试代码（消除重复、无硬编码 ID）

### 3.7 P1-7：View 层数据注解 + ViewModel 强类型

**当前问题表现**：

**面试展示价值**：中。代码整洁度的加分项，但不是亮点。

**AI 可参与的工作点**：

**关键产物**：

- 所有 ViewModel + Entity 添加 Display 注解

### 3.8 P1-8：数据库迁移 + 跨平台运行能力

**当前问题表现**：
- `EnsureCreated` 每次启动重建 Schema，无法增量迁移
- 依赖 SQL Server LocalDB（仅 Windows），macOS/Linux 用户无法运行演示
- README 写"仅 Windows"，面试官如果使用 Mac 就无法现场演示

**面试展示价值**：高。"我配置了 EF Core 迁移 + SQLite 兼容，可以在任何平台上 3 步跑起来"是很有说服力的工程话术。

**商业化价值**：跨平台 = 更大的用户基础。

**AI 可参与的工作点**：
- 生成 `dotnet ef migrations add InitialCreate` 迁移代码
- 配置 Program.cs 根据环境切换 SQL Server / SQLite
- 修改 README 启动步骤增加 SQLite 选项
- 生成跨平台测试方案

**建议开发步骤**：
1. `dotnet ef migrations add InitialCreate` 生成首次迁移
2. 修改 Program.cs：`UseSqlServer` 或 `UseSqlite` 根据配置切换
3. `appsettings.json` 增加 SQLite 连接字符串 `"SqliteConnection": "Data Source=libraryseat.db"`
4. 将启动时 `EnsureCreated` 替换为 `Database.Migrate()`
5. 修改 README 启动步骤，增加 SQLite 选项
6. 验证在 macOS/Linux 上可以 `dotnet run` 并正常使用

**关键产物**：
- `Migrations/` 目录 + 初始迁移文件
- Program.cs 数据库提供者切换逻辑
- appsettings.json 增加 SQLite 连接字符串
- 更新 README 跨平台启动说明

**最低验收线**：
- `dotnet ef migrations` 可正常运行
- 设置 SQLite 连接字符串后应用可在任意平台启动
- 所有 Service 功能在 SQLite 下正常（特别注意 `DateTime` 和 `TimeSpan` 的行为差异）
- 测试可以针对 SQLite 运行（解决 LocalDB 仅在 Windows 的限制）

## 5. 不建议现在就投入的过度设计项

| 项目 | 原因 |
|------|------|
| 用户注册功能 | 课堂项目无注册需求，增加复杂度但不增加展示价值 |
| OAuth2/第三方登录 | 严重过度设计，面试官反而会觉得"你为什么不先做好基础密码认证" |
| SignalR 实时刷新 | 预约冲突检测已由 AJAX 处理完够。在 9 页项目中引入 WebSocket 是炫技 |
| Redis 缓存 | 数据量极小，缓存带来的复杂度远大于收益。面试时会暴露"没有想清楚就引入" |
| 微服务拆分 | 致命过度设计。单人项目拆微服务纯粹是技术债 |
| Docker 容器化 | 有价值但优先级低。LocalDB 无法容器化，拆 SQLite 后可以再考虑 |
| 性能压测/负载测试 | 15 个座位的系统不需要，面试官不会期待 |
| 多语言国际化 | 项目面向中文用户，加英文资源文件是纯额外工作量 |
| 前端框架替换（React/Vue） | ASP.NET Core MVC + jQuery 是合理的组合。换前端框架是重写不是升级 |

## 6. 最终结论

### 如果只继续打磨一轮（时间有限），第一优先级做什么？

**选 P0-1（管理员密码认证）+ P0-2（site.js 提取）**。

理由：
- 密码认证是"课堂项目"和"可展示产品"之间最明显的分界线
- site.js 提取是立竿见影的代码组织改善，5 个 View 清空内联 JS 后项目代码感完全不同
- 这两项加在一起，如果集中精力 + AI 辅助，1-2 天可以完成

### 最能让面试官感受到项目成熟度的 3 个证据

| 证据 | 为什么有效 |
|------|-----------|
| **管理员登录需要密码** | 面试官现场演示时看到密码输入框 + 错误提示，马上会意识到"这个项目的安全不是摆设" |
| **GitHub 仓库有 CI 绿色徽章** | 视觉信号比语言描述强 10 倍。README 顶部一个绿色 build passing 徽章 + "test: 30 passed" 是最直接的工程化证据 |
| **View 源代码干净** | 面试官如果打开 GitHub 上的 .cshtml 文件，看到的是结构清晰的 Razor 标签 + 一行 `@section Scripts { <script src="~/js/site.js"></script> }`，没有大段 `$(function() { ... })` 堆在文件中间 |

这三个证据覆盖了三个层面：**安全意识**（密码）、**工程化**（CI）、**代码质量**（JS 组织）。每一项都可以在面试中展开讲 1-2 分钟。
