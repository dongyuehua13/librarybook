# 图书馆座位预约系统 - 项目立项单

## 1. 项目基本信息

- **项目名称**：图书馆座位预约系统
- **开发者**：软件专业实训学生（单人独立完成）
- **开发周期**：课程实训周期（约 4-5 周）
- **技术栈**：ASP.NET Core MVC (.NET 8) + Razor 视图 + SQL Server LocalDB + Entity Framework Core + Bootstrap
- **交付物**：GitHub 或 Gitee 完整代码仓库，含 README 启动说明
- **运行方式**：clone → `dotnet run` → 浏览器打开，无需额外配置

## 2. 项目背景

高校图书馆自习座位一直是个老大难问题：学生大清早去排长队占座，结果占了座人又不来，空座浪费严重，管理员靠纸笔登记效率低得可怜。

这次做一个轻量级的网页版座位预约系统，解决"找座难、占座乱、管座累"这三个基本问题。注意这是单人课堂实训项目，不做企业级复杂功能，目标是做一个能跑、能演示、代码拿得出手的小系统。

## 3. 目标用户

| 角色 | 说明 | 系统入口 |
|------|------|---------|
| **在校学生** | 想在图书馆自习，需要查座位、预约、取消预约 | 用户端（前台） |
| **图书馆管理员** | 负责维护座位信息、查看预约情况、统计数据 | 管理端（后台） |

就这两类用户，没有第三类。不用考虑超级管理员、不用考虑访客。

## 4. 项目目标

1. 学生能在线看到图书馆有哪些座位空着，选一个时段预约上，不想去了能自己取消；
2. 管理员能登录后台新增座位、下架坏掉的座位、看到谁预约了哪个座位、座位的使用率怎么样；
3. 网页用 Bootstrap 做响应式布局，手机和电脑上都能看，不用额外装 App；
4. 数据存本地 SQL Server LocalDB，不用连远程服务器，代码拉到本地一键就能跑起来。

## 5. 本期必须完成内容

### 5.1 用户端（学生用）

| # | 页面/功能 | 做什么用的 |
|---|----------|-----------|
| 1 | 首页 | 展示总座位数、今日预约数，提供各功能入口 |
| 2 | 座位列表 | 展示所有座位，区分"空闲/已预约"状态，支持按楼层和区域筛选 |
| 3 | 座位详情 | 点进单个座位看详细信息（位置、描述），以及该座位的预约记录 |
| 4 | 预约提交 | 选日期、选开始/结束时段，提交预约请求，系统做时段冲突检查 |
| 5 | 我的预约 | 当前登录学生的所有预约记录列表 |
| 6 | 取消预约 | 在"我的预约"页面一键取消还没开始的预约 |
| 7 | 登录方案 | 不搞注册登录，直接下拉选学生账号切换，演示够用就行 |

### 5.2 管理端（管理员用）

| # | 页面/功能 | 做什么用的 |
|---|----------|-----------|
| 1 | 管理员登录 | Session + Cookie 方式做轻量登录校验 |
| 2 | 座位管理 | 新增座位、编辑座位信息、禁用/启用座位 |
| 3 | 预约管理 | 查看所有学生的全部预约记录，支持按状态和日期筛选 |
| 4 | 数据统计 | 展示座位预约热度排行、近 14 日预约趋势（用进度条做简易图表） |

### 5.3 底层基础

- EF Core 三张数据库表：`Users`、`Seats`、`Reservations`，做好关联关系
- 管理员和学生的种子数据（预设 1 个管理员 + 3 个学生 + 15 个座位）
- Bootstrap 响应式布局，页面看着像那么回事

## 6. 本期暂不开发内容

以下功能课堂阶段明确不做，别往项目里加：

1. **微信扫码登录** — 涉及第三方平台，课时内搞不定
2. **短信/消息推送** — 通知功能，超出单人工作量
3. **在线支付、押金、违规扣费** — 自习场景不需要钱
4. **学生注册/密码找回/权限分级** — 模拟账号解决一切
5. **多图书馆、多楼层分区联动管理** — 单楼层够演示
6. **超时自动释放座位** — 需要后台定时任务，不做
7. **人脸识别核验** — 需要硬件，纯 Web 做不了
8. **导出 Excel、打印报表** — 手工抄数据够用

## 7. 页面规模与页面分布

一共 **9 个页面**（账号切换嵌入导航栏，不单独计页），单人开发量适中：

**用户端（6 页）**：首页 → 座位列表 → 座位详情 → 预约提交 → 我的预约 → 切换账号

**管理端（4 页）**：管理员登录 → 座位管理 → 预约管理 → 数据统计

每个页面就是一个 `.cshtml` 视图文件，对应一个控制器 Action。

## 8. 关键业务主流程

**流程一：学生预约座位**
> 打开系统 → 首页看个大概 → 进座位列表（可按楼层筛） → 点一个空闲座位看详情 → 点"预约"选日期和时段 → 提交 → 去"我的预约"确认

**流程二：学生取消预约**
> 进"我的预约" → 找到还没开始的预约 → 点"取消" → 确认 → 状态变"已取消"

**流程三：管理员管理后台**
> 进后台登录页 → 输入管理员账号 → 进座位管理增删改 → 进预约管理查记录 → 进统计页看数据

## 9. 成功标准（现场演示时验证）

演示时要能做到以下动作：

1. **启动项目**：在命令行执行 `dotnet run`，浏览器输入地址，能正常打开首页，所有页面没有 404；
2. **学生完整流程**：点"切换账号"选一个学生 → 浏览座位列表 → 查看座位详情 → 选时段提交预约 → 在"我的预约"看到新记录 → 点击取消，预约状态变为"已取消"；
3. **管理员操作**：访问 `/Admin/Login` → 输入 `admin` 登录 → 新增一个座位 → 编辑座位信息 → 禁用某个座位 → 在预约管理页看到刚才学生创建的预约记录；
4. **数据统计展示**：统计页能看到座位热度排行和预约趋势图（用进度条画的也算），数据是活的不是写死的；
5. **代码可复现**：把代码推到 GitHub/Gitee，另一个人 clone 下来执行 `dotnet run` 就能跑起来，不需要额外装数据库或配环境。

## 10. 风险与假设

### 风险（课堂真实情况）

| 风险 | 可能性 | 应对措施 |
|------|--------|---------|
| 时间不够用，页面做到一半就到截止日期 | 高 | MVP 先做核心流程，体验功能往后放 |
| EF Core 关联查询报错，卡在 ORM 上 | 中 | 提前写好三张表关系，不搞复杂查询 |
| AI 生成的代码有隐藏 bug，调试花很多时间 | 中 | 每写完一个功能立即测试，不堆在一起排错 |
| Bootstrap 样式调来调去还是丑 | 中 | 用现成组件，不自己写 CSS，保证对齐就行 |
| 页面写太多，单人忙不过来 | 高 | 严格卡在 9 页范围内，绝不加页 |

### 假设（按这些假设来做）

1. **单楼层**：只管理一栋图书馆的座位数据，无需分馆分区；
2. **预约即生效**：学生提交预约直接成功，不需要管理员审核；
3. **无惩罚机制**：学生取消预约没有次数限制，不做黑名单；
4. **本地开发**：全程用 LocalDB，不部署远程服务器；
5. **一人完成**：没有团队协作，一切自己搞定，范围自己控制。
