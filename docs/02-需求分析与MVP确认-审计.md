# 《图书馆座位预约系统》需求分析与MVP确认 审计报告

## A. 总体判断

文档质量较高，6项检查要点全部达标：

1. **需求分析真实到位**：从立项单出发，完成了用户分类、场景描述、痛点排序、候选功能池构建，不是立项单的简单改写，而是真正的需求拆解；
2. **功能优先级矩阵明确**：P0/P1/P2/P3 四级划分清晰，每个功能都标注了归类理由，可追溯可复核；
3. **MVP 范围确认明确**：第5节明确圈定 P0+P1 共17项功能、10个页面，附关键流程验证和工作量估算；
4. **删减项/暂缓项完整**：第7节完整列出保留项(7)、删减项(8)、暂缓项(4)，每项均标注删减/暂缓原因；
5. **需求评审结论成立**：范围收束逻辑自洽，剔除的功能有合理依据，MVP 范围经得起推敲；
6. **最终范围适合单人课堂**：10个页面、估算5天，与项目立项单口径一致，工作量可控。

**结论：文档成熟度较高，不存在阻断性问题，可直接进入 PRD Lite 阶段。**

## B. 发现的问题清单

| 编号 | 问题类型 | 问题描述 | 位置 |
|------|----------|----------|------|
| 1 | 细化不足 | 工作量估算仅到模块级（0.5+2+1.5+1=5天），未与立项单中3个里程碑关联，无法直接用于进度跟踪 | 第5.3节 |
| 2 | 缺失 | 无非功能性需求描述（页面加载时间、浏览器兼容最低版本、并发量等），常规实训虽不严格要求，但作为完整需求文档建议补充 | 全文缺失 |
| 3 | 细化不足 | 无正式的 User Story 格式（As a... I want... so that...），当前场景描述虽已覆盖核心流程，但缺少可独立验收的粒度 | 第1.2节 |
| 4 | 缺失 | 无单个功能的验收标准，仅在第8节有整体测试标准，"每个功能怎样算做完"未逐一定义 | 第8.3节 |
| 5 | 可优化 | 数据统计页实现方式未明确（立项单建议"极简图表"），可进一步确认是以表格+CSS进度条实现还是引入 Chart.js | 第4.1节 A04 |

## C. 必改项

**无强制必改项。** 需求分析完整、MVP 范围清晰、结论成立，所有问题均为建议优化级别，不修改也可进入 PRD Lite。

## D. 建议优化项

| 编号 | 建议内容 | 关联问题 |
|------|----------|----------|
| 1 | **工作量估算与里程碑绑定**：将 5 天估算拆分到立项单的3个里程碑——M1: 项目初始化+数据层+页面骨架(1天)；M2: 用户端6页开发(2天)；M3: 管理端5页+联调+上传(2天) | 问题1 |
| 2 | **补充非功能性需求（一句话即可）**：例如"支持 Chrome/Firefox/Edge 最新版浏览器，页面加载不超过3秒，无需部署服务器即可本地演示" | 问题2 |
| 3 | **核心功能补充验收标准**：例如"预约提交——选择日期时段后点击提交，若时段无冲突则提示成功并跳转到我的预约；若冲突则提示错误保留表单" | 问题4 |
| 4 | **明确统计页实现方案**：在 MVP 范围或优先级矩阵中注明——"数据统计采用 Bootstrap 进度条 + 表格展示，不引入第三方图表库" | 问题5 |

## E. 是否可以进入下一步"PRD Lite"

**可以。** 需求分析充分、MVP 范围收敛合理、工作量可控、风险可控，满足进入 PRD Lite 阶段的所有前置条件。
