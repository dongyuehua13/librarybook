# 图书馆座位预约系统 - 数据库设计

> 文档版本：v1.0  
> 前置输入：docs/03-PRD-Lite.md、docs/04-页面树与业务流程.md、docs/07-系统设计说明.md  
> 下一步产出：关键链路详细设计

---

## 1. 数据设计目标

1. **支撑核心业务流程** — 学生查询座位、提交预约、取消预约，管理员管理座位、查看预约和统计。
2. **保证数据一致性** — 同一座位同一日期同一时段只能有一个有效预约，通过数据库和应用层双重保障。
3. **简化下沉到 EF Core** — 三张表，关系简单，Code First 自动建库，种子数据预设。
4. **查询效率够用** — 课堂项目数据量小（百级），不做分表分库，不做索引调优。

---

## 2. 核心实体清单

以前序文档（PRD-Lite §11）已确认的 3 个实体为准，不做额外拆分。

| 实体 | 表名 | 说明 |
|------|------|------|
| 用户 | `Users` | 学生和管理员账号，区分角色用 `IsAdmin` 字段 |
| 座位 | `Seats` | 自习座位，含楼层/区域/启用状态 |
| 预约记录 | `Reservations` | 学生对座位的时段预约，含日期/时段/状态 |

**不拆分的理由：**
- 不拆分 `SeatArea` 独立表：楼层+区域作为 Seats 的普通字段，筛选用 WHERE 查询，不需要建主外键。
- 不拆分 `TimeSlot` 独立表：开始和结束时间直接存在 Reservations 中，不做预定义的时段表。
- 不拆分 `ReservationLog`：取消行为通过 Status 字段记录，不额外建操作日志表。

## 3. 关系说明

### 3.1 实体关系

```
Users (1) ────── (N) Reservations (N) ────── (1) Seats
  │                      │                       │
  │                      │  UserId (FK)          │
  │                      │  SeatId (FK)          │
  │                      │                       │
  └── Id (PK)            └──                     └── Id (PK)
      Username                                        SeatNumber
      DisplayName                                      Floor
      IsAdmin                                          Area
                                                       Description
                                                       IsActive
```

### 3.2 关系描述

| 关系 | 类型 | 外键 | 说明 |
|------|------|------|------|
| Users → Reservations | 一对多 | Reservation.UserId → User.Id | 一个学生可以有多条预约记录 |
| Seats → Reservations | 一对多 | Reservation.SeatId → Seat.Id | 一个座位可以被多次预约（不同日期或时段） |

### 3.3 关系约束

- 删除用户时，不删除预约记录（保留历史数据）
- 删除座位时，不删除预约记录（保留历史数据）
- 禁用座位时，不影响已有预约记录

## 4. 表结构设计

### 4.1 Users（用户）

| 字段名 | 数据类型 | 约束 | 默认值 | 说明 |
|--------|---------|------|--------|------|
| Id | `int` | PK, 自增 | — | 用户 ID |
| Username | `nvarchar(50)` | NOT NULL, UNIQUE | — | 登录名，管理员用此登录 |
| DisplayName | `nvarchar(100)` | NOT NULL | — | 显示名称，页面展示用 |
| IsAdmin | `bit` | NOT NULL | `false` | 是否管理员：true=管理员，false=学生 |

**EF Core 映射：**
```csharp
public class User
{
    public int Id { get; set; }
    public string Username { get; set; }
    public string DisplayName { get; set; }
    public bool IsAdmin { get; set; }
    public List<Reservation> Reservations { get; set; } = new();
}
```

### 4.2 Seats（座位）

| 字段名 | 数据类型 | 约束 | 默认值 | 说明 |
|--------|---------|------|--------|------|
| Id | `int` | PK, 自增 | — | 座位 ID |
| SeatNumber | `nvarchar(50)` | NOT NULL | — | 座位编号，如 A-01、B-03 |
| Floor | `int` | NOT NULL | `1` | 所在楼层 |
| Area | `nvarchar(50)` | NOT NULL | `'自习区'` | 区域名称：自习区/安静区/电子阅览区 |
| Description | `nvarchar(200)` | NULL | — | 位置描述：靠窗、有插座等 |
| IsActive | `bit` | NOT NULL | `true` | 是否启用：true=启用(可预约)，false=禁用(下架) |

**EF Core 映射：**
```csharp
public class Seat
{
    public int Id { get; set; }
    public string SeatNumber { get; set; }
    public int Floor { get; set; }
    public string Area { get; set; }
    public string? Description { get; set; }
    public bool IsActive { get; set; }
    public List<Reservation> Reservations { get; set; } = new();
}
```

### 4.3 Reservations（预约记录）

| 字段名 | 数据类型 | 约束 | 默认值 | 说明 |
|--------|---------|------|--------|------|
| Id | `int` | PK, 自增 | — | 预约 ID |
| UserId | `int` | FK → Users.Id, NOT NULL | — | 预约用户 |
| SeatId | `int` | FK → Seats.Id, NOT NULL | — | 预约座位 |
| Date | `date` | NOT NULL | — | 预约日期（不含时间） |
| StartTime | `time(0)` | NOT NULL | — | 开始时段，如 `09:00` |
| EndTime | `time(0)` | NOT NULL | — | 结束时段，如 `11:00` |
| Status | `nvarchar(20)` | NOT NULL | `'已预约'` | 状态：已预约 / 已取消 |
| CreatedAt | `datetime2(3)` | NOT NULL | `GETDATE()` | 预约创建时间 |

**EF Core 映射：**
```csharp
public class Reservation
{
    public int Id { get; set; }
    public int UserId { get; set; }
    public User User { get; set; } = null!;
    public int SeatId { get; set; }
    public Seat Seat { get; set; } = null!;
    public DateTime Date { get; set; }
    public TimeSpan StartTime { get; set; }
    public TimeSpan EndTime { get; set; }
    public string Status { get; set; } = "已预约";
    public DateTime CreatedAt { get; set; } = DateTime.Now;
}
```

### 4.4 DbContext 关系配置

```csharp
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    // Users → Reservations (一对多)
    modelBuilder.Entity<Reservation>()
        .HasOne(r => r.User)
        .WithMany(u => u.Reservations)
        .HasForeignKey(r => r.UserId)
        .OnDelete(DeleteBehavior.NoAction);  // 不级联删除

    // Seats → Reservations (一对多)
    modelBuilder.Entity<Reservation>()
        .HasOne(r => r.Seat)
        .WithMany(s => s.Reservations)
        .HasForeignKey(r => r.SeatId)
        .OnDelete(DeleteBehavior.NoAction);  // 不级联删除

    // Username 唯一索引
    modelBuilder.Entity<User>()
        .HasIndex(u => u.Username)
        .IsUnique();
}
```

## 5. 关键字段说明

| 字段 | 设计考虑 | 特别注意 |
|------|---------|---------|
| `User.IsAdmin` | 不建 Role 表，用 bit 区分。系统只有"管理员/学生"两级 | 不要设计成 int RoleId 关联 Role 表，用不上 |
| `Seat.SeatNumber` | 字符串，不自动生成，由管理员手工输入 | 不拆成 Building + Floor + Number 三段 |
| `Seat.Floor` | int，直接用数字，不用字符串"1楼" | 前端显示时拼接"楼"字 |
| `Seat.Area` | 自由字符串，不做外键关联 | 筛选时用 WHERE Area = @area |
| `Reservation.Date` | 用 `date` 类型，不含时间 | 不要用 `datetime` 存，日期查询会慢 |
| `Reservation.StartTime` / `EndTime` | 用 `time(0)` 类型，精确到分钟 | 不用 `datetime`，方便做时段重叠计算 |
| `Reservation.Status` | 用字符串而非 int 枚举 | 代码可读性好，EF Core 不需要额外转换配置 |
| `Reservation.CreatedAt` | 用 `datetime2(3)` 精确到毫秒 | 排序靠此字段 |

## 6. 状态字段设计

### 6.1 预约状态 (Reservation.Status)

| 状态值 | 含义 | 进入条件 | 允许后续操作 |
|--------|------|---------|------------|
| `已预约` | 预约有效，座位被占用 | 学生提交预约，无时段冲突 | → 已取消 |
| `已取消` | 预约作废，座位释放 | 学生在可取消时间内点击取消 | 无（终态） |

**设计说明：**
- 使用字符串而非 int Enum：`Status = "已预约"` 在代码中直接可读，不需要写 Enum.Parse
- 仅两个状态，不做"待审核""已完成""爽约"等中间状态
- 已取消的记录保留在表中用于统计用途

### 6.2 座位状态 (Seat.IsActive)

| 状态值 | 含义 | 前台表现 |
|--------|------|---------|
| `true`（启用） | 座位可用，可被预约 | 在座位列表中显示 |
| `false`（禁用） | 座位不可用，已下架 | 不在座位列表中显示 |

**设计说明：**
- 使用 bit 而非字符串：无需中间状态，启用/禁用是二值逻辑
- 禁用不影响已有预约记录（已预约的继续有效）

## 7. 查询与筛选字段设计

### 7.1 前台查询

| 场景 | 查询条件 | 对应 SQL/EF 写法 |
|------|---------|-----------------|
| 座位列表（无筛选） | `IsActive == true` | `.Where(s => s.IsActive)` |
| 按楼层筛选 | `IsActive == true && Floor == floor` | `.Where(s => s.IsActive && s.Floor == floor)` |
| 按区域筛选 | `IsActive == true && Area == area` | `.Where(s => s.IsActive && s.Area == area)` |
| 当日预约座位 ID 集合 | `Date == today && Status == "已预约"` | `.Where(r => r.Date == today && r.Status == "已预约").Select(r => r.SeatId)` |
| 座位预约记录 | `SeatId == id && Date >= today` | `.Where(r => r.SeatId == id && r.Date >= today).OrderBy(r => r.Date)` |
| 我的预约 | `UserId == userId` | `.Where(r => r.UserId == userId).OrderByDescending(r => r.CreatedAt)` |
| 时段冲突检测 | `SeatId == seatId && Date == date && Status == "已预约" && start < EndTime && end > StartTime` | `.Any(r => r.SeatId == seatId && r.Date == date && r.Status == "已预约" && startTime < r.EndTime && endTime > r.StartTime)` |

### 7.2 后台查询

| 场景 | 查询条件 | 对应 SQL/EF 写法 |
|------|---------|-----------------|
| 预约管理（全部） | 无筛选 | `.Include(r => r.User).Include(r => r.Seat).OrderByDescending(r => r.CreatedAt)` |
| 按状态筛选 | `Status == status` | `.Where(r => r.Status == status)` |
| 按日期范围筛选 | `Date >= startDate && Date <= endDate` | `.Where(r => r.Date >= startDate && r.Date <= endDate)` |
| 座位管理（全部） | 无筛选 | `.OrderBy(s => s.SeatNumber)` |
| 单座位查询 | `Id == id` | `.FirstOrDefault(s => s.Id == id)` |

### 7.3 筛选字段索引建议

| 表 | 字段 | 索引类型 | 理由 |
|----|------|---------|------|
| Reservations | `(SeatId, Date, Status)` | 复合索引 | 时段冲突检测高频查询 |
| Reservations | `(UserId, Date)` | 复合索引 | 我的预约高频查询 |
| Reservations | `(Date, Status)` | 复合索引 | 管理员日期+状态筛选 |
| Seats | `(Floor, Area, IsActive)` | 复合索引 | 座位列表筛选查询 |

**课堂项目说明：** 数据量极小（百级），索引非必需。以上索引设计供参考，开发阶段不建索引也不影响演示。如果使用 EF Core Code First 自动建库，默认不会建这些索引，需要的话在 `OnModelCreating` 中手动加。

## 8. 统计字段与统计口径

### 8.1 统计口径定义

| 统计项 | SQL 口径 | 说明 |
|--------|---------|------|
| 总可用座位数 | `SELECT COUNT(*) FROM Seats WHERE IsActive = 1` | 前台首页和统计页共用 |
| 总预约次数 | `SELECT COUNT(*) FROM Reservations` | 包含已预约和已取消 |
| 今日预约数 | `SELECT COUNT(*) FROM Reservations WHERE Date = CAST(GETDATE() AS DATE) AND Status = N'已预约'` | 当日有效预约数（仅已预约） |
| 座位预约热度排行 | `SELECT SeatId, COUNT(*) as Cnt FROM Reservations WHERE Status = N'已预约' GROUP BY SeatId ORDER BY Cnt DESC` | 只统计有效预约 |
| 近 14 日预约趋势 | `SELECT Date, COUNT(*) as Cnt FROM Reservations WHERE Date >= DATEADD(DAY, -13, CAST(GETDATE() AS DATE)) GROUP BY Date ORDER BY Date` | 含今天往前 14 天 |

### 8.2 统计服务接口

```csharp
public class DashboardStats
{
    public int TotalActiveSeats { get; set; }
    public int TodayReservations { get; set; }
}

public class AdminStats
{
    public int TotalActiveSeats { get; set; }
    public int TotalReservations { get; set; }
    public int TodayReservations { get; set; }
    public List<SeatRanking> SeatRankings { get; set; }
    public List<DailyTrend> DailyTrends { get; set; }
}

public class SeatRanking
{
    public string SeatNumber { get; set; }
    public int Count { get; set; }
}

public class DailyTrend
{
    public DateTime Date { get; set; }
    public int Count { get; set; }
}
```

### 8.3 统计实现说明

- 所有统计在服务端完成，Service 层做聚合查询，Controller 传入 ViewBag 或 ViewModel
- 不做前端 JS 二次聚合
- 热度排行的最大值为 100% 基准，其他按比例计算百分比

## 9. 数据一致性与约束说明

### 9.1 时段冲突约束

同一座位同一日期同一时段不可有两个有效预约。

**SQL 冲突检测条件：**
```sql
WHERE SeatId = @seatId
  AND Date = @date
  AND Status = N'已预约'
  AND @startTime < EndTime
  AND @endTime > StartTime
```

如果查到任何记录，拒绝预约。

**边界情况处理：**
- 开始时间 = 前一条的结束时间 → 不冲突（如 09:00-11:00 和 11:00-13:00 不重叠）
- 结束时间 > 开始时间 → 在前端和后端都做校验

### 9.2 外键约束

| 约束 | 类型 | 违反后果 |
|------|------|---------|
| Reservation.UserId → Users.Id | FK | 插入不存在的 UserId 报错 |
| Reservation.SeatId → Seats.Id | FK | 插入不存在的 SeatId 报错 |
| User.Username | UNIQUE | 插入重复 Username 报错 |

### 9.3 业务层约束（不在数据库层实施）

| 约束 | 实施位置 | 说明 |
|------|---------|------|
| 仅 "已预约" 状态可取消 | Service.Cancel() | 校验 Status 值 |
| 过期预约不可取消 | Service.Cancel() | 校验 Date 和 StartTime 与当前时间 |
| 仅管理员可管理座位 | AdminController 的 Session 校验 | 每个 Action 入口检查 AdminId |
| 学生只能操作自己的预约 | Service.GetUserReservations() | 通过 UserId 过滤 |

### 9.4 不做数据库约束

| 约束 | 不做的理由 |
|------|-----------|
| 预约时间不可重叠的 CHECK 约束 | 跨行约束无法用 CHECK 实现，需用表级约束或触发器，EF Core 不适合 |
| 开始时间 < 结束时间的 CHECK 约束 | 前端 + 后端双重校验即可，数据库级约束过度设计 |
| Status 的 CHECK 约束 | 字符串类型，C# 业务层控制 |

## 10. 当前阶段实现边界

| 维度 | 本阶段实现 | 不考虑 |
|------|-----------|--------|
| 表数量 | 3 张（Users, Seats, Reservations） | 日志表、操作记录表、审批表 |
| 数据量 | 预设 4 个用户 + 15 个座位 | 万级数据、分页查询 |
| 软删除 | 不做，直接物理删除 | — |
| 事务 | 单表操作，EF Core 默认事务 | 跨表事务、分布式事务 |
| 索引 | 不建，数据量小不影响 | 复合索引、覆盖索引 |
| 视图/存储过程 | 不用，EF Core 直接写 LINQ | 复杂 SQL 视图、存储过程 |
| 级联删除 | 关闭，保留历史数据 | ON DELETE CASCADE |
| 数据库迁移 | 使用 `EnsureCreated()` 自动建库 | EF Migration、版本回滚 |

### 11.3 设计决策记录

| 决策 | 选项 | 选择 | 理由 |
|------|------|------|------|
| 楼层和区域是否独立表 | 独立 / 不独立 | 不独立 | 仅做筛选标签，无独立业务逻辑 |
| 预约状态用 int 还是 string | int Enum / string | string | EF Core 直接存储可读字符串，方便查看数据库 |
| 座位状态用 bit 还是 string | bit / string | bit | 二值逻辑，bit 更省空间且语义明确 |

| 是否建 ReservationLog | 建 / 不建 | 不建 | 取消通过 Status 表示，无需额外日志表 |
