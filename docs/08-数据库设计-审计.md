# 《图书馆座位预约系统》数据库设计 审计报告

## A. 总体判断

10 项检查要点全部达标：

1. **实体/字段/状态与前序文档一致**：User、Seat、Reservation 三表的全部字段命名、类型、状态值与 docs/03 §11、docs/04 §6、docs/07 §3 完全对齐，无偏差；
2. **未擅自删平为固定 6 实体**：严格保留前序文档的 3 实体骨架，Section 11 列出了不引入 SeatArea/TimeSlot/ReservationLog 的具体理由，合理有据；
3. **表关系合理**：User 1:N Reservation、Seat 1:N Reservation，ER 图清楚，外键约束明确；
4. **支撑页面树和业务流程**：9 页的数据读写与 4 条核心流程在索引一节有明确的查询映射，无遗漏；
5. **状态字段清楚**：Reservation.Status 两态（已预约→已取消）和 Seat.IsActive 两态（启用↔禁用）均有状态机图、触发条件和前台表现说明；
6. **查询与筛选字段充分**：楼层筛选、区域筛选、状态筛选、日期范围筛选在索引建议和 SQL WHERE 中有完整映射；
7. **统计字段与口径明确**：3 项统计卡片、热度排行（含百分比公式）、14 日趋势均有精确的 SQL 实现和数据一致性说明；
8. **覆盖预约冲突规则**：Section 9 列出了 6 条应用层约束（时段冲突、重复预约、时长、每日上限、座位可用性、取消时效），并给出了冲突检测的 SQL 原型；
9. **默认补足项显式写出且合理**：Section 11 列出 6 项补充决策，每一项都标注了理由；
10. **适合 LocalDB + EF Core + 课堂项目**：3 表设计、无过度拆分、无存储过程/视图/触发器、无复杂并发控制，适合单人 MVC 项目。

## B. 发现的问题清单

| 编号 | 问题类型 | 问题描述 | 位置 |
|------|----------|----------|------|
| 1 | 缺失 | 索引建议使用了 SQL Server 的 INCLUDE 语法，但 EF Core Code First 默认不生成 INCLUDE 索引，需要 Fluent API 或原始 SQL 迁移，课堂项目可能不会实际执行 | §7.1 |
| 2 | 可优化 | `Users.Username` 标注了 UNIQUE 约束，但代码中未通过 `[Index]` 或 Fluent API 显式配置，仅靠种子数据保证唯一 | §4.1 |
| 3 | 可优化 | 统计口径中"总预约次数"包含已取消记录，但热度排行仅统计 Status='已预约'，口径不一致可能造成理解偏差 | §8.1 vs §8.2 |
