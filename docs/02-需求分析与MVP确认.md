## 1. 目标用户与核心场景

### 1.1 用户分类

| 用户角色 | 身份说明 | 系统入口 | 典型频率 |
|---------|---------|---------|---------|
| 在校学生 | 本校本科生/研究生，需使用自习座位 | 用户端（前台） | 每日1-3次 |
| 图书馆管理员 | 图书馆工作人员，负责座位与预约管理 | 管理端（后台） | 每日1-2次 |

### 1.2 核心场景描述

**场景一：学生查看空闲座位并完成预约**
> 大三学生张明早上8点来到图书馆，想找一个自习座位。他打开座位预约系统，浏览座位列表，看到3楼电子阅览区C-03座位目前空闲。他点进详情页确认座位配有电源插座，选择"今天 9:00-12:00"时段提交预约。系统提示预约成功，他到馆直接入座。

**场景二：学生因故取消预约**
> 张明临时接到导师通知要开会，无法按计划去图书馆。他打开"我的预约"页面，找到今天9:00-12:00的预约记录，点击取消按钮，系统将该预约状态变为"已取消"，座位C-03释放供他人预约。

**场景三：管理员维护座位信息**
> 管理员李老师发现1楼A-02座位台灯损坏，暂时无法使用。他登录管理后台，在座位管理页面找到A-02，将其状态改为"禁用"，防止学生误预约。维修完成后再次启用。

**场景四：管理员查看使用数据**
> 图书馆馆长需要了解本周座位使用情况。管理员登录后台进入数据统计页，查看近14日预约趋势图和座位预约热度排行，发现2楼安静区座位预约率最高，据此调整各区座位配置。

## 2. 用户最关心的问题

### 2.1 学生用户

| 排序 | 问题 | 影响程度 |
|------|------|---------|
| 1 | 能不能快速看到哪些座位现在空闲？ | 高 — 决定系统是否可用 |
| 2 | 选了时段会不会跟别人冲突？ | 高 — 预约信任度 |
| 3 | 操作是否简单，手机上能用吗？ | 中 — 使用意愿 |
| 4 | 预约了去不了能不能取消？ | 高 — 避免违规担忧 |
| 5 | 要不要注册/下载App？ | 中 — 使用门槛 |

### 2.2 管理员用户

| 排序 | 问题 | 影响程度 |
|------|------|---------|
| 1 | 能不能方便地新增/禁用座位？ | 高 — 日常管理效率 |
| 2 | 能不能查到谁预约了哪个座位？ | 高 — 管理核心诉求 |
| 3 | 能不能统计座位利用率？ | 中 — 管理决策支撑 |
| 4 | 需不需要装额外软件？ | 中 — 推广成本 |

## 5. MVP 范围确认

### 5.1 MVP 包含功能（P0 + P1）

| 模块 | 功能清单 | 页面数 |
|------|---------|--------|
| **用户端** | 首页、座位列表（含筛选）、座位详情、预约提交、我的预约、取消预约、模拟账号切换 | 6 |
| **管理端** | 管理员登录/注销、座位管理（新增/编辑/禁用）、预约管理（查看/筛选）、数据统计 | 5 |
| **底层** | 三张核心表、LocalDB持久化、种子数据、自动建库 | — |
| **合计** | 17 项功能，覆盖全部核心流程 | 9 个页面 |

### 5.2 MVP 关键流程验证

```
流程1（学生预约）：切换账号 → 首页 → 座位列表(筛选楼层) → 座位详情 → 选择时段提交 → 我的预约
流程2（学生取消）：我的预约 → 点击取消 → 确认 → 状态变更为已取消
流程3（管理员管理）：登录后台 → 座位管理(新增/编辑/禁用) → 预约管理(按状态筛选) → 数据统计
```

### 5.3 MVP 工作量估算

| 模块 | 估算人天（单人） | 说明 |
|------|----------------|------|
| 项目初始化 + 数据层 | 0.5 天 | 脚手架搭建、模型配置、种子数据 |
| 用户端开发 | 2 天 | 6个页面 + 控制器逻辑 |
| 管理端开发 | 1.5 天 | 5个页面 + 控制器逻辑 |
| 样式调试 + 联调 | 1 天 | Bootstrap响应式适配、全流程测试 |
| **合计** | **5 天** | 符合课堂实训周期 |

## 6. 本期明确不做的功能

以下功能在项目立项单及候选功能池中已识别，但在 MVP 阶段明确排除，理由说明如下：

| 功能 | 排除理由 |
|------|---------|
| 用户注册/密码找回 | 立项单已明确采用模拟账号切换，无需完整注册体系 |
| 微信扫码登录 | 立项单暂不开发内容，涉及第三方对接 |
| 短信/消息推送 | 立项单暂不开发内容，远超课堂单人工作量 |
| 在线支付/押金 | 立项单暂不开发内容，与自习场景无直接关系 |
| 违规扣费/黑名单 | 立项单假设已简化规则，不做复杂惩罚逻辑 |
| 多图书馆/多楼层联动 | 立项单假设仅需单楼层，且筛选已满足楼层需求 |
| 超时自动释放座位 | 立项单暂不开发内容，需定时任务/后台服务 |
| 人脸识别核验 | 立项单暂不开发内容，涉及硬件设备对接 |
| 批量导入/导出Excel | 立项单暂不开发内容，手动录入演示即可 |
| 预约审核流程 | 简化后采用直接预约模式，审核增加流程负担 |
| 多级权限体系 | 仅学生/管理员两级，符合课堂项目规格 |

## 8. 进入 PRD Lite 前的最终口径

### 8.1 一句话定位
> 面向高校学生的轻量化图书馆座位预约系统，支持在线查看空闲座位、预约时段、自主取消预约，并提供管理员后台进行座位维护与使用数据统计。

### 8.2 核心设计约束
1. **单楼层单区域数据模型** — 不做多馆多区联动
2. **模拟账号切换** — 不做完整账号体系
3. **直接预约无需审核** — 预约即生效，简化流程
4. **Bootstrap 响应式** — 适配PC与移动端浏览器，不做原生App
5. **无第三方服务依赖** — 纯LocalDB + 本地运行，无需联网

### 8.3 交付物口径
| 维度 | 口径 |
|------|------|
| 功能范围 | 10个页面，17项功能（P0+P1），覆盖完整预约+管理流程 |
| 技术栈 | ASP.NET Core MVC 8 + Razor + EF Core + LocalDB + Bootstrap |
| 测试标准 | 本地启动 → 模拟学生完成预约/取消 → 管理员完成座位管理/统计查看 |
| 交付形式 | Gitee/GitHub 仓库，含完整代码，README 写明启动步骤 |
| 不做事项 | 注册体系、消息推送、在线支付、自动化规则、报表导出 |

1. 基于本确认文档，编写 `docs/03-PRD-Lite.md`（含功能规格、页面原型描述、数据字典）
