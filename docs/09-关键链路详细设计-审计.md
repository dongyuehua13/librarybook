# 《图书馆座位预约系统》关键链路详细设计 审计报告

## A. 总体判断

10 项检查要点全部达标：

1. **用户/管理链路沿用前序文档**：3 条链路的步骤命名、顺序、职责与 docs/03 §8 核心流程和 docs/04 §4/§5 步骤拆解完全一致，无擅自重命名、改步骤或改职责；
2. **用户链路完整闭环**：首页→座位列表→详情→预约提交→我的预约→返回首页，路径闭合，取消链路也在我的预约页内自闭环；
3. **管理链路完整闭环**：登录→座位管理(CRUD)→预约管理→统计→退出→回到登录页，路径闭合；
4. **路由清楚**：第4节职责矩阵完整列出 18 个路由与 Controller.Action 的一一映射，含 Session 校验/数据读写/业务规则标注；
5. **分工合理**：Controller 负责参数接收/Session 校验/页面跳转，DbContext 负责数据读写，业务校验内联在 Action 中（Section 8 已写明不抽 Service 层的理由，合理）；
6. **数据库变化与 docs/08 对应**：第5节查询/写入清单仅操作 Users/Seats/Reservations 三表，字段名与 docs/08 §4 完全一致，无额外表或字段引入；
7. **状态流转明确且一致**：第6节对照表清晰列出 Reservation（—→已预约→已取消）和 Seat（true↔false）的完整流转，与 docs/08 §6 状态机完全一致；
8. **异常与边界考虑充分**：第7节覆盖 18 个异常场景，3 条链路各自有独立的异常处理表，包括未登录/时段冲突/每日上限/已过期/已开始/无效 ID 等；
9. **默认补足项显式写出且合理**：Section 8 列出 4 项补充决策（Controller 内联逻辑、ViewBag 传统计、5 次独立查询、View 内判断取消按钮），均有理由说明；
10. **足以支撑代码实现**：每条链路的每一步均有路由/Controller Action/Session 校验/数据查询 SQL/数据写入/异常返回的完整规格，可直接编码。

## B. 发现的问题清单

| 编号 | 问题类型 | 问题描述 | 位置 |
|------|----------|----------|------|
| 1 | 可优化 | 冲突检测使用了 5 次独立数据库查询（时长/时段/上限/座位冲突/用户重复），可合并为 2-3 次以降低数据库往返 | §1.2 步骤④ |
| 2 | 可优化 | 统计页的热度排行查询在 Service 中使用了嵌套 LINQ（`Count(r => r.SeatId == s.Id)`），在 EF Core 中可能翻译为 N+1 查询，建议改用 GROUP BY + JOIN | §3.2 步骤⑤ |
| 3 | 缺失 | 未提及种子数据中的初始预约记录——演示时"今日预约 3"和预约记录列表需要预设数据 | 全文 |

## C. 必改项

**无强制必改项。** 关键链路设计完整、与上游文档一致、异常覆盖充分，不存在硬性缺陷。

## D. 建议优化项

| 编号 | 建议内容 | 关联问题 |
|------|----------|----------|
| 1 | 冲突检测的 5 次查询可合并为：① 时长+时段合法性（Service 层计算，不查库）② 每日计数 1 次 ③ 座位冲突+用户重复可合并为 1 次查询带 OR 条件 | 问题1 |
| 2 | 热度排行改用 `_db.Reservations.Where(r => r.Status == "已预约").GroupBy(r => r.SeatId).Select(g => new { SeatId = g.Key, Count = g.Count() })` 避免 N+1 | 问题2 |
| 3 | 在 DbInitializer 中补充几条预约演示数据，使"今日预约 3"等统计数字在首次启动时有展示效果 | 问题3 |
