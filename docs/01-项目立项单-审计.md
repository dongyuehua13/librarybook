# 《图书馆座位预约系统》项目立项单 审计报告

## A. 总体判断

立项单完整覆盖11个章节，6项审计核心要点基本达标：

1. **项目目标清楚**：明确了学生预约、管理员后台管理两大核心价值，方向清晰无歧义；
2. **本期范围与边界清楚**：第5节"本期必须完成内容"与第6节"本期暂不开发内容"划分明确，无范围模糊地带；
3. **页面规模适合单人课堂项目**：合计10个页面（用户端6 + 管理端4），工作量适配单人实训周期；
4. **成功标准具体可演示**：5条成功标准全部为可现场操作演示、可逐条核对验收的硬性指标；
5. **风险与假设真实**：4条风险贴合学生实训真实痛点，4条假设贴合本地简易开发环境；
6. **里程碑数量合理**：仅设置3个里程碑，不存在过多、过散问题。

整体无阻断项目推进的严重缺陷，可直接进入下一阶段。

## B. 发现的问题清单

| 编号 | 问题类型 | 问题描述 | 位置 |
|------|----------|----------|------|
| 1 | 细化不足 | 项目目标无量化业务约束，未说明单次预约时长、单日预约上限等规则，目标描述偏笼统 | 第4节 |
| 2 | 细化不足 | 关键业务流程仅描述正常操作流程，缺少预约冲突、重复预约等异常场景说明 | 第8节 |
| 3 | 细化不足 | 风险与假设仅罗列内容，未配套对应的风险规避、解决办法 | 第10节 |
| 4 | 细化不足 | 里程碑仅写明开发内容，无预估课时/工作日划分，无法直观把控实训进度 | 第11节 |
| 5 | 细化不足 | 页面规模部分仅罗列页面数量，未简单说明每个页面的核心功能定位 | 第7节 |

## C. 必改项

**无强制必改项。** 立项单核心范围、验收标准、项目边界全部完整，不存在会导致无法开展需求分析的硬性缺陷，不修改也可进入下一阶段。

## D. 建议优化项

| 编号 | 建议内容 | 关联问题 |
|------|----------|----------|
| 1 | **项目目标补充量化规则**：学生单日最多预约2个时段，单次预约最长4小时，避免目标过于笼统 | 问题1 |
| 2 | **完善业务流程异常分支**：增加"重复预约同一时段，系统弹出提示并拦截提交"、"所选时段已过有效期"等异常流程说明 | 问题2 |
| 3 | **为每条风险补充对应缓解方案**：<br>- 风险①实训时间不足 → 缓解：优先完成用户端预约核心流程，统计页面采用极简图表<br>- 风险②EF Core数据库关联报错 → 缓解：分阶段建表，先完成单表CRUD再做联表查询<br>- 风险③AI生成代码存在兼容性bug → 缓解：分段生成代码，每写完一段本地运行测试<br>- 风险④Bootstrap页面布局错乱 → 缓解：复用基础模板，减少自定义样式开发 | 问题3 |
| 4 | **每个里程碑补充预估课时**：例如里程碑1预估6课时、里程碑2预估8课时、里程碑3预估6课时，均衡拆分单人开发工作量 | 问题4 |
| 5 | **在页面规模模块简单标注每个页面的核心用途**：<br>- 首页：座位总数统计 + 系统入口导航<br>- 座位列表：楼层/区域筛选，展示占用状态<br>- 座位详情：单座位信息 + 可选时段<br>- 预约提交：日期时段选择 + 提交预约<br>- 我的预约：本人预约列表 + 取消操作<br>- 管理登录：Session校验<br>- 座位管理：CRUD座位<br>- 预约管理：查看/筛选全部预约<br>- 统计页：使用率+每日预约量 | 问题5 |

## E. 是否可以进入下一步"需求分析与 MVP 确认"

**可以。** 立项单范围可控、验收标准明确、工作量适配单人开发，满足阶段准入全部要求，可直接开展需求拆解与MVP最小产品确认工作。
